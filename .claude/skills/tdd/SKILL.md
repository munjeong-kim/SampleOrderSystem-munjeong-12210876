# TDD 기반 구현 워크플로우

S-Semi 시료 생산주문관리 시스템은 Python + pytest + pytest-mock 기반으로,
모든 기능 구현은 Red-Green-Refactor 사이클을 따른다.

## 절차

1. **Red — 계획 작성 + 실패하는 테스트 작성**
   - 구현하려는 기능/요구사항을 루트의 `PLAN.md`에 먼저 작성하거나 업데이트한다.
   - 해당 요구사항에 대한 테스트를 `tests/` 하위에 작성한다.
   - 외부 의존성(파일 입출력, 입력 프롬프트 등)은 `pytest-mock`(`mocker` fixture)으로 모킹한다.
   - 테스트를 실행하여 실패(또는 수집 오류)를 확인한다.
   - `PLAN.md`와 테스트 코드를 `[RED]` 말머리로 커밋한다. (예: `[RED] 주문 생성 기능 테스트 추가`)
   - 커밋 후 사용자에게 Red 단계 결과(PLAN.md 내용, 작성한 테스트, 실패 확인 결과)를 보여주고 컨펌을 받는다. 승인 전에는 Green 단계로 넘어가지 않는다.

2. **Green — 테스트를 통과시키는 최소 구현**
   - 테스트를 통과시키는 데 필요한 최소한의 코드만 작성한다.
   - 과도한 설계나 미래를 위한 추상화는 넣지 않는다.
   - 테스트를 실행하여 통과를 확인한다.
   - 구현 코드를 `[Green]` 말머리로 커밋한다. (예: `[Green] 주문 생성 기능 구현`)
   - 커밋 후 사용자에게 Green 단계 결과(구현 내용, 테스트 통과 결과)를 보여주고 컨펌을 받는다. 승인 전에는 Refactor 단계로 넘어가지 않는다.

3. **Refactor — 리팩터링**
   - 테스트가 통과하는 상태를 유지하면서 중복 제거, 이름 개선 등 코드 정리를 한다.
   - 리팩터링 후 다시 테스트를 실행해 회귀가 없는지 확인한다.
   - 리팩터링 결과를 `[Refactor]` 말머리로 커밋한다. (리팩터링할 내용이 없으면 이 단계와 커밋은 생략 가능)
   - 커밋 후(또는 리팩터링할 내용이 없다고 판단한 경우) 사용자에게 결과를 보여주고 컨펌을 받은 뒤 다음 요구사항의 Red 단계로 넘어간다.

4. 기능 단위가 끝날 때마다 전체 테스트(`pytest`)를 실행해 기존 기능이 깨지지 않았는지 확인한다.

## 규칙

- 테스트 없이 프로덕션 코드를 먼저 작성하지 않는다.
- 한 번에 하나의 작은 요구사항(테스트 하나)에 집중한다.
- 테스트 파일명은 `test_*.py`, 함수명은 `test_*` 컨벤션을 따른다.
- 각 단계(Red/Green/Refactor)마다 반드시 커밋을 나누고, 커밋 메시지 말머리로 `[RED]`, `[Green]`, `[Refactor]`를 붙인다.
- 여러 단계의 변경을 하나의 커밋에 섞지 않는다.
- 각 단계의 커밋이 끝나면 반드시 사용자에게 컨펌을 받은 뒤에만 다음 단계로 진행한다. 컨펌 없이 연속으로 여러 단계를 진행하지 않는다.
